# Phase 9-C Fix 1: Repository Hygiene and Version Fingerprint Stability

## المشكلة

كانت بصمة نسخة المصدر تتضمن `checksum` و`size_bytes`. هاتان القيمتان تظهران
عند التنزيل، لكنهما قد تغيبان عند فحص البيانات الوصفية فقط. لذلك كان المصدر
نفسه يُسجل أحيانًا كنسخة جديدة لمجرد الانتقال بين `download=true` و
`download=false`.

## الإصلاح

أصبحت هوية النسخة تعتمد أولًا على:

- هوية المصدر.
- الهوية الخارجية.
- ETag.
- تاريخ التعديل الخارجي.

ويستخدم SHA-256 والحجم كبديل فقط عندما لا يوفر المزود ETag أو تاريخ تعديل.
كما يعاد استخدام الإصدار المرجعي عند ثبات البيانات الخارجية، ثم يُثرى لاحقًا
بالمسار المحلي وSHA-256 عند تنزيل الملف.

## اختبارات الانحدار

يغطي الإصلاح الاتجاهين:

1. تنزيل الملف ثم فحص البيانات الوصفية فقط.
2. فحص البيانات الوصفية ثم تنزيل الملف.

في الحالتين يبقى سجل النسخ مكونًا من نسخة واحدة، وتضاف بيانات التحقق إلى
السجل نفسه دون إنشاء نسخة معلقة كاذبة.

## نظافة المستودع

لا تتضمن النسخة الكاملة النظيفة الملف المولد:

`backend/data/madarik.sqlite3`

ولا تضع حزمة الملفات المعدلة أي ملف مساعد في جذر المستودع.
