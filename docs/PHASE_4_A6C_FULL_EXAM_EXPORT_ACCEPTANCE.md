# Phase 4-A6c: Full Exam Export Acceptance

## Purpose

Phase 4-A6c closes the full-exam workflow by validating the **actual DOCX and PDF artifacts** produced after intake, translation, and teacher review.

The phase does not merely assume that a successful renderer call means the paper is structurally safe. Each generated file receives an invisible, deterministic structure manifest and is reopened for inspection before an acceptance result is persisted in the project snapshot.

## Scope

The phase verifies:

- active question count and export order;
- hierarchy-safe mark totals without counting parent and child marks twice;
- multipart question count;
- question image attachments inserted once;
- valid DOCX/PDF container structure;
- PDF page availability and text layer;
- source-page linkage and multi-page question counts retained in the report;
- completion of all formats selected in `ProjectMetadata.export_formats`;
- upstream full-exam intake and translation acceptance state.

The page snapshots linked during Phase 4-A6a remain traceability references. A6c exports cropped question assets and explicit attachments, not entire PDF pages, avoiding duplicated full-page scans beside already reconstructed question text.

## Artifact manifest

Both export engines embed an ASCII metadata manifest with the prefix:

```text
MADARIK-A6C-V1
```

It records:

- question count;
- structured part count;
- attachment count;
- hierarchy-safe total marks;
- display order.

DOCX stores the manifest in the core `subject` property. PDF stores it in document metadata. The acceptance service reads it back from the generated bytes and compares it with the current project.

## Persisted models

`ProjectSession.full_exam_export_report` stores:

- overall state: `accepted`, `needs_review`, `incomplete`, or `failed`;
- requested and generated formats;
- accepted, review, and failed formats;
- expected questions, parts, attachments, and marks;
- source-page and multi-page traceability counts;
- one detailed artifact summary per format;
- global checks and warnings.

The report is invalidated when metadata, questions, translation, ordering, source links, logo, or question attachments change.

## API behavior

The existing endpoints still return downloadable binary files:

```text
POST /api/projects/{project_id}/export/docx
POST /api/projects/{project_id}/export/pdf
```

After each generation they persist the artifact inspection result and return:

```text
X-Madarik-Export-Acceptance: accepted|needs_review|incomplete|failed
```

The current report is available from:

```text
GET /api/projects/{project_id}/export/acceptance
```

## Acceptance semantics

- `incomplete`: at least one selected format has not yet been generated.
- `accepted`: every selected format exists, every artifact check passes, and upstream full-exam intake and translation reports are accepted.
- `needs_review`: files are usable but a warning exists, such as an invalid attachment, total-mark mismatch, or missing upstream acceptance.
- `failed`: the generated file is not a readable DOCX/PDF artifact, its manifest is absent, or there are no active questions.

## UI

The export step displays:

- overall artifact acceptance state;
- generated versus requested formats;
- questions, marks, parts, attachments, and source linkage;
- per-format size, PDF page count, exported structure, checks, and warnings.

The frontend reloads the project after each binary download so the persisted report appears immediately.

## Tests

`test_full_exam_export_acceptance.py` covers:

1. DOCX remains `incomplete` until the requested PDF is generated.
2. Combined DOCX/PDF acceptance with multipart hierarchy, source pages, and one attachment.
3. Upstream translation acceptance requirement.
4. Invalid-image warning without blocking file generation.
5. API persistence and combined report retrieval.
