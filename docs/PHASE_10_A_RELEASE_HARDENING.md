# Phase 10-A: Release Hardening and End-to-End Acceptance

## الهدف

إغلاق البوابة التقنية للإصدار دون إدخال ميزات تعليمية جديدة أو إعادة هيكلة
واسعة. تركز المرحلة على كشف الأعطال قبل الدمج، وتوحيد حالة الجاهزية، وإثبات
أن المسارات المحلية الرئيسية تعمل معًا.

## نطاق التنفيذ

### 1. بيانات الإصدار

أضيف مصدر مركزي داخل:

`backend/app/core/release.py`

ويحتوي الإصدار البرمجي وقناة الإصدار واسم المرحلة. يستخدم FastAPI المصدر نفسه
بدل رقم قديم منفصل.

### 2. فحص الجاهزية الآمن

أضيف المسار:

`GET /api/health/readiness`

ويفحص:

- فتح قاعدة SQLite وسلامة `PRAGMA quick_check`.
- وجود الجداول الأساسية التسعة.
- قابلية إنشاء مجلد البيانات والكتابة فيه.
- قابلية إنشاء مجلد التصدير والكتابة فيه.
- وضع مزود الترجمة واكتمال الإعداد عند تفعيل الاتصال الخارجي.
- وضع Google Drive وOneDrive واكتمال إعداد المزود عند تفعيله.

لا يعرض المسار مفاتيح أو رموز وصول أو أسرار OneDrive أو مسارات قاعدة البيانات
ومجلدات الخادم.

### 3. حالات الجاهزية

- `ready`: لا موانع ولا تحذيرات.
- `degraded`: البوابة التقنية ناجحة، مع مزود اختياري معطل أو قبول حي معلق.
- `blocked`: يوجد فشل مطلوب، مثل مخطط قاعدة ناقص أو مزود خارجي مفعل بإعداد ناقص.

الحقل `technical_ready` هو الحكم التقني المباشر. لا يعني نجاحه أن الإصدار
الإنتاجي اعتمد، لأن القبول الحي والمراجعة البصرية ما زالا مستقلين.

### 4. تدقيق المستودع

أضيف:

`RUN_PHASE10_A_RELEASE_AUDIT.py`

ويتحقق من:

- عدم تتبع SQLite وZIP وPatch وملفات Python المولدة.
- عدم تتبع ملفات أسرار وشهادات ومفاتيح خاصة.
- عدم وجود مفاتيح شائعة داخل الملفات النصية.
- تطابق الإصدار بين README وBackend و`package.json` و`package-lock.json`.
- عدم رجوع الملفات المبعثرة إلى جذر المستودع.
- وجود ملفات المرحلة ووثائقها.
- عدم تكرار `operationId` في OpenAPI.
- ظهور `/api/health/readiness` في OpenAPI ونجاح فحصه دون تسريب حقول حساسة.

### 5. اختبار القبول المحلي الشامل

يغطي الاختبار الجديد رحلة متصلة تشمل:

1. إنشاء مشروع.
2. رفع PDF نصي واستخراج النص.
3. تقسيم الأسئلة.
4. توليد القاموس.
5. الترجمة المحلية الآمنة.
6. جاهزية المشروع.
7. تصدير DOCX وPDF.
8. حفظ سؤال في بنك الأسئلة.
9. إنشاء مسودة اختبار وإضافة السؤال.
10. توليد ثلاثة أنشطة متمايزة وتصدير نشاط PDF.
11. إنشاء رسم علمي وتصديره PNG وPDF.
12. تحديث مصدر OneDrive وهمي، اعتماد النسخة الأولى، وإدخالها إلى مشروع جديد.
13. نجاح تقرير جاهزية الإصدار على قاعدة الاختبار نفسها.

لا يجري الاختبار اتصالًا حيًا بـMicrosoft Graph أو Gemini أو OpenAI.

### 6. GitHub Actions

تعمل البوابة عند:

- الرفع إلى `main`.
- الرفع إلى `feat/madarik-science-platform-v2`.
- فتح Pull Request إلى `main`.
- التشغيل اليدوي.

وتشغل:

- تدقيق Phase 10-A.
- `compileall`.
- اختبارات Phase 10-A المركزة.
- اختبارات Backend الكاملة.
- `npm ci`.
- Frontend lint.
- Frontend production build.
- `git diff --check` النهائي.

## خارج النطاق

- لا ترجمة حية بمزود خارجي داخل CI.
- لا إنشاء Tag أو GitHub Release.
- لا اعتماد إنتاجي دون مراجعة DOCX وPDF بصريًا.
- لا تغيير في عقود المشاريع أو الأسئلة أو المصادر السحابية.
- لا إعادة تصميم للواجهة.

## معيار الإغلاق التقني

تغلق Phase 10-A تقنيًا بعد نجاح الاختبارات المحلية وGitHub Actions. يبقى
الإصدار الإنتاجي محجوبًا حتى استكمال العناصر الحية في قائمة القبول النهائية.
