# Phase 4-A6d: Full Exam End-to-End Acceptance Gate

## الهدف

تضيف هذه المرحلة بوابة قبول واحدة تجمع نتائج المراحل السابقة:

1. قبول إدخال الورقة الكاملة.
2. تغطية لقطات صفحات PDF وروابطها بالأسئلة.
3. اعتماد القاموس العلمي.
4. قبول الترجمة والمراجعة.
5. جاهزية التصدير.
6. إنشاء DOCX وفحصه في الذاكرة.
7. إنشاء PDF وفحصه في الذاكرة.
8. فحص الاتساق النهائي بين أعداد الأسئلة والدرجات والصيغ.

البوابة لا تعيد تشغيل مزود الذكاء الاصطناعي، ولا تستبدل ترجمة راجعها
المعلم، ولا تنزل الملفات تلقائيًا. وظيفتها فحص الحالة الحالية للمشروع
وإنشاء ملفات مؤقتة في الذاكرة لاختبار سلامتها.

## الحالات النهائية

- `accepted`: جميع المراحل المطلوبة وصلت إلى حالة قبول.
- `needs_review`: لا يوجد فشل مانع، لكن توجد مرحلة ناقصة أو تحتاج مراجعة.
- `rejected`: توجد ملاحظة جاهزية مانعة أو فشل في مرحلة مطلوبة.

## حالات المراحل

- `accepted`
- `needs_review`
- `pending`
- `failed`
- `skipped`

تظهر صيغة التصدير بحالة `skipped` عندما لا يطلبها المستخدم في إعدادات
المشروع، ولا يخفض ذلك حالة البوابة النهائية.

## النماذج الجديدة

- `FullExamEndToEndReport`
- `FullExamEndToEndStageSummary`
- `FullExamEndToEndCheck`
- `FullExamEndToEndAcceptanceStatus`
- `FullExamEndToEndStageStatus`
- `FullExamEndToEndStageKey`

يحفظ التقرير:

- معرف تشغيل مستقل.
- وقت إنشاء التقرير.
- الزمن الإجمالي وزمن كل مرحلة بالمللي ثانية.
- عدد الصفحات والأسئلة ومجموع الدرجات.
- نسبة اكتمال الترجمة.
- الصيغ المطلوبة والمنشأة والمقبولة.
- نتائج كل مرحلة وتحذيراتها وأخطائها.

## الخدمة

الملف:

```text
backend/app/services/full_exam_end_to_end.py
```

الدالة العامة:

```python
run_full_exam_end_to_end_acceptance(project)
```

تعيد الدالة:

- تقرير القبول الشامل.
- تقرير الإدخال المعاد حسابه.
- تقرير الترجمة المعاد حسابه.
- تقرير الجاهزية.
- تقرير التصدير الناتج من فحص الملفات الفعلية.

تعمل الدالة على نسخة عميقة من المشروع، فلا تعدل المحتوى الأصلي أثناء
الفحص.

## API

### تشغيل البوابة

```http
POST /api/projects/{project_id}/full-exam/acceptance/run
```

يعيد `ProjectSession` بعد حفظ التقارير المتسقة.

### قراءة آخر تقرير

```http
GET /api/projects/{project_id}/full-exam/acceptance
```

يعيد التقرير المحفوظ أو `null` إذا لم تُشغّل البوابة أو تغير المشروع بعد
آخر تشغيل.

## إبطال التقرير

يُلغى تقرير A6d تلقائيًا عندما يتغير ما يؤثر في النتيجة، مثل:

- بيانات المشروع.
- الملف أو النص المستخرج.
- الأسئلة أو ترتيبها أو حالتها.
- القاموس.
- لقطات الصفحات وروابطها.
- الترجمة.
- مرفقات الأسئلة.
- تقرير تصدير جديد.

هذا يمنع بقاء شارة قبول قديمة فوق مشروع تغيرت بنيته، وهي عادة بشرية
سيئة بما يكفي خارج البرمجيات.

- الحالة النهائية.
- عدد الصفحات والأسئلة والدرجات.
- نسبة اكتمال الترجمة.
- حالة كل مرحلة وزمنها.

## الاختبارات

الملف:

```text
backend/tests/test_full_exam_end_to_end_acceptance.py
```

يغطي خمسة سيناريوهات:

1. قبول مشروع كامل مع DOCX وPDF.
2. قبول مشروع يطلب DOCX فقط مع تجاوز PDF.
3. تحويل نقص اعتماد المعلم إلى `needs_review`.
4. رفض مشروع يحتوي أخطاء جاهزية مانعة.
5. حفظ تقرير API ثم إبطاله بعد تعديل سؤال.

العدد المتوقع بعد المرحلة:

```text
218 passed
```

## خارج النطاق

- تشغيل الترجمة آليًا من داخل البوابة.
- رفع PDF جديد من زر البوابة.
- تنزيل Word وPDF تلقائيًا.
- OCR جديد أو تحسين محلل الأسئلة.
- تعديل محتوى المعلم أو القاموس بصمت.

تنتقل إلى الرحلة الكاملة الآن نتائج حارس تغطية الرسومات لكل سؤال، وحارس تعارض الدرجة المعلنة مع مجموع الأسئلة. يبقى إنشاء الملف متاحًا للمراجعة، لكن لا تصل الرحلة إلى `accepted` عند فقد رسم مطلوب أو ظهور تعارض مثل 20 مقابل 80.
