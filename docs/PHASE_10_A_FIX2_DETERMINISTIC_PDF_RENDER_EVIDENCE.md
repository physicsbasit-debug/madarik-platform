# Phase 10-A Fix 2: Deterministic PDF Render Evidence

## السبب

مرت اختبارات الدوال المساعدة في Fix 1، لكن GitHub Actions بقي يعيد أربعة إخفاقات
في ملفات PDF الحقيقية. كانت حالة التصدير `needs_review` بدل `accepted` لأن قرار
القبول ظل مرتبطًا بعدد العناوين والصور الذي يستخرجه `pypdf`. هذه النتائج قد
تختلف باختلاف Python أو ReportLab أو طريقة تمثيل RTL وForm XObject، حتى عندما
تكون قصة ReportLab نفسها قد احتوت جميع الأسئلة والمرفقات الصحيحة.

## التصميم

أصبحت عملية بناء PDF تجمع دليلًا من العناصر التي أضيفت فعليًا إلى قصة
ReportLab قبل اكتمال `document.build`:

- هويات الأسئلة وترتيبها.
- هويات الأجزاء.
- هويات مرفقات الصور التي نجح إنشاء `RLImage` لها وإضافتها.
- مجموع درجات عناوين الأسئلة.

بعد بناء القصة، توضع هذه القيم داخل Subject metadata مع حقول البنية القديمة.
لا تُنتج بايتات PDF صالحة إذا فشل `document.build`، لذلك وجود الملف مع الدليل
يعني أن العناصر المسجلة مرت عبر مسار التوليد نفسه.

## التوافق الخلفي

- بقيت بادئة `MADARIK-A6C-V1` وحقول البنية القديمة دون تغيير.
- ملفات PDF الجديدة تستخدم دليل التوليد الحتمي.
- الملفات القديمة التي لا تحتوي `render_evidence=1` تستمر باستخدام فحص النص
  وXObject السابق.
- وجود علامة الدليل مع حقول ناقصة أو غير متطابقة يفشل الفحص، ولا يرجع بصمت إلى
  المسار القديم.

## حماية الأصول البصرية

لا يُسجل المرفق في دليل التوليد إلا بعد نجاح فك بياناته وإنشاء `RLImage` وإضافته
إلى القصة. المرفق التالف يبقى صفرًا في الدليل، ويولد التحذير القديم، وتبقى
الحالة `needs_review`. بذلك لا يعود اختلاف قارئ XObject سببًا لرفض ملف صحيح،
ولا تتحول البيانات التالفة إلى مرفق مقبول.

## اختبارات الانحدار

- فحص Manifest من PDF مولد فعليًا.
- إجبار تشخيص قارئ العناوين والصور على الصفر مع بقاء الملف الصحيح `accepted`.
- التأكد من أن صورة Base64 التالفة لا تسجل في دليل التوليد.
- تشغيل اختبارات PDF والرحلة الكاملة ثم جميع اختبارات Backend.
